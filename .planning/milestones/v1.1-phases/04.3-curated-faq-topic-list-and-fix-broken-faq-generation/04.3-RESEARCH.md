# Phase 04.3: Curated FAQ Topic List & Fix Broken FAQ Generation - Research

**Researched:** 2026-03-12
**Domain:** LLM pipeline refactoring, JSON output robustness, curated content management
**Confidence:** HIGH

## Summary

The FAQ generation pipeline is broken due to a token limit overflow. The current `generateFAQ()` function sends the entire topic taxonomy (~200KB prompt) to Claude in a single call, producing a response that exceeds the 32K output token limit (34,923 tokens attempted). The response is truncated mid-JSON, starting with a closing `}` fragment that even `repairJSON()` cannot recover from. The debug output confirms `stop_reason: "end_turn"` with `maxOutputTokens: 32000` and `outputTokens: 34923`.

The fix requires splitting FAQ generation into per-topic LLM calls using a curated topic list as the authoritative input. There are 1,759 unique topic labels across all 23 law summaries, which is far too many for direct FAQ generation. A curated list of 12-18 key municipal governance themes will serve as the stable input, with each topic generating its own LLM call that receives only the relevant paragraph references. This naturally stays within token limits (each topic produces ~2-4KB JSON).

**Primary recommendation:** Create `data/llm/faq/curated-topics.json` with 12-18 curated topics, refactor `generateFAQ()` to iterate per-topic with filtered paragraph data, aggregate results into existing `topics.json` format for backward compatibility with `generate-pages.js`.

## Architecture Patterns

### Recommended File Structure
```
data/llm/faq/
  curated-topics.json       # NEW: Authoritative curated topic list (version-controlled)
  topics/                   # NEW: Per-topic generated results (optional, for granular regeneration)
    sitzungsoeffenlichkeit.json
    befangenheit.json
    ...
  topics.json               # EXISTING: Aggregated FAQ output (consumed by generate-pages.js)
```

### Pattern 1: Curated Topic List as Authoritative Input
**What:** A hand-maintained JSON file that defines the FAQ topic space, decoupled from LLM serendipity.
**When to use:** Always -- this is the input to FAQ generation.
**Schema:**
```json
{
  "version": 1,
  "topics": [
    {
      "slug": "sitzungsoeffentlichkeit",
      "title": "Sitzungsoeffentlichkeit und Vertraulichkeit",
      "description": "Oeffentlichkeit von Gemeinderatssitzungen, Ausschluss der Oeffentlichkeit, Vertraulichkeit",
      "seedQuestions": [
        "Sind Gemeinderatssitzungen oeffentlich?",
        "Wann kann die Oeffentlichkeit ausgeschlossen werden?"
      ]
    }
  ]
}
```

### Pattern 2: Per-Topic LLM Generation
**What:** Each curated topic gets its own LLM call with only the relevant paragraph references.
**When to use:** For all FAQ generation.
**How it works:**
1. Load curated topics from `curated-topics.json`
2. For each topic, collect matching paragraphs from summaries (filter by topic label overlap)
3. Build a focused prompt with topic metadata + filtered references
4. Call Claude once per topic
5. Aggregate all per-topic results into `topics.json`

**Key detail:** Filtering paragraphs to a topic requires matching curated topic slugs/titles against the 1,759 topic labels in summaries. Use substring matching or keyword overlap since curated titles are broader categories that encompass many specific labels.

### Pattern 3: Backward-Compatible Output
**What:** The final `topics.json` must match the existing schema consumed by `generate-pages.js`.
**Schema (unchanged):**
```json
{
  "meta": { "generatedAt": "...", "topicCount": 15 },
  "topics": [
    {
      "slug": "...",
      "title": "...",
      "description": "...",
      "questions": [
        {
          "question": "...",
          "answer": "...",
          "references": [{ "category": "...", "key": "...", "paragraph": "...", "label": "..." }]
        }
      ]
    }
  ]
}
```
**Consumers:** `generate-pages.js` lines 784-810 read `topics.json` and generate:
- `src/faq/index.html` (FAQ index with topic cards)
- `src/faq/{slug}.html` (individual topic pages)

### Anti-Patterns to Avoid
- **Sending all 1,759 topics in a single prompt:** This is the current bug. Even with truncation to 100 topics, the response exceeds token limits.
- **Relying on LLM to discover topics:** Without curated input, each regeneration produces different topics, making the FAQ section unstable.
- **Modifying `callClaude()`:** This function is shared across summaries, FAQ, and glossary. Changes here risk regressions.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| JSON repair | Custom parser | Existing `repairJSON()` | Already handles unescaped quotes; per-topic calls avoid truncation |
| Topic matching | Exact string match | Keyword overlap / substring matching | Curated topics are broad categories; summary labels are highly specific |
| Prompt templating | Template engine | Template literals | Existing pattern in codebase; prompts are simple string interpolation |

## Common Pitfalls

### Pitfall 1: Topic-to-Summary Matching is Lossy
**What goes wrong:** Curated topic "Befangenheit" must match summary labels like "Befangenheit", "Befangenheitsregelungen", "Befangenheit Gemeinderat", "Befangenheit und Ausschluss", etc.
**Why it happens:** 1,759 unique labels with inconsistent naming across 23 laws.
**How to avoid:** Use case-insensitive substring matching: if any summary topic label contains the curated topic's slug words, include that paragraph. Also consider the curated topic description for additional keywords.
**Warning signs:** A curated topic matches zero paragraphs -- means matching logic is too strict.

### Pitfall 2: Per-Topic Prompt Still Too Large
**What goes wrong:** A popular topic like "Gemeinderatssitzungen" matches hundreds of paragraphs across 23 laws.
**Why it happens:** Some themes appear in nearly every law.
**How to avoid:** Cap references per topic (e.g., max 6 per law, ~150 total). Current code already does `refs.slice(0, 6)` per topic -- apply similar logic per-topic.
**Warning signs:** Prompt exceeds 50KB for a single topic.

### Pitfall 3: Gendering and Style Instructions Missing from Per-Topic Prompts
**What goes wrong:** Per-topic FAQ answers lack gendering, use BL-specific claims, or have ASCII-safe spellings.
**Why it happens:** Splitting the prompt means the style instructions must be included in every per-topic prompt.
**How to avoid:** Extract the gendering/style instructions into a constant string, include in every per-topic prompt.

### Pitfall 4: Race Condition with Incremental Regeneration
**What goes wrong:** If one topic fails mid-generation, `topics.json` may contain stale data from a previous run mixed with new per-topic results.
**Why it happens:** Aggregation happens after all topics are generated.
**How to avoid:** Either regenerate all topics or clearly track which per-topic files are current. The `--force` flag should clean all per-topic results before regenerating.

### Pitfall 5: Existing Tests Assume Single-Call FAQ Generation
**What goes wrong:** Test at `tests/llm-analyze.test.js` line 94 checks "FAQ prompt does NOT hardcode topic list" and line 101 checks "FAQ prompt instructs Claude to determine topics."
**Why it happens:** The new approach uses a curated topic list, which changes the prompt structure.
**How to avoid:** Update tests to reflect new architecture: curated topics as input, per-topic generation, etc.

### Filtering Paragraphs for a Curated Topic
```javascript
// Source: New pattern based on existing collectTopicTaxonomy + topicParagraphs logic
function collectParagraphsForTopic(curatedTopic, rootDir = ROOT) {
  const keywords = [
    curatedTopic.slug.replace(/-/g, ' '),
    curatedTopic.title.toLowerCase(),
    // Extract key words from description
    ...curatedTopic.description.toLowerCase().split(/[,\s]+/).filter(w => w.length > 4)
  ];

  const matches = [];
  for (const category of CATEGORIES) {
    const summaryDir = join(rootDir, 'data', 'llm', 'summaries', category);
    if (!existsSync(summaryDir)) continue;
    for (const file of readdirSync(summaryDir).filter(f => f.endsWith('.json'))) {
      const lawKey = file.replace('.json', '');
      const data = JSON.parse(readFileSync(join(summaryDir, file), 'utf-8'));
      if (!data.paragraphs) continue;
      for (const [paraNum, para] of Object.entries(data.paragraphs)) {
        if (!para.topics) continue;
        const topicStr = para.topics.join(' ').toLowerCase();
        if (keywords.some(kw => topicStr.includes(kw))) {
          matches.push({ category, key: lawKey, paragraph: paraNum, summary: para.summary });
        }
      }
    }
  }
  return matches;
}
```

### Per-Topic LLM Call
```javascript
// Source: Adapted from existing generateFAQ prompt pattern
function buildTopicPrompt(curatedTopic, paragraphRefs) {
  const citationRef = Object.entries(BL_CITATION)
    .map(([key, abbr]) => `  ${key} = "${abbr}"`)
    .join('\n');

  // Cap refs: max 6 per law key to keep prompt manageable
  const byLaw = {};
  for (const ref of paragraphRefs) {
    const k = `${ref.category}/${ref.key}`;
    if (!byLaw[k]) byLaw[k] = [];
    if (byLaw[k].length < 6) byLaw[k].push(ref);
  }
  const cappedRefs = Object.values(byLaw).flat();

  const refText = cappedRefs.map(r => {
    const citation = BL_CITATION[r.key] || r.key;
    return `- Par. ${r.paragraph} ${citation}: ${r.summary?.substring(0, 120)}`;
  }).join('\n');

  return `Du bist ein Experte fuer oesterreichisches Gemeinderecht.

Erstelle FAQ-Eintraege zum Thema "${curatedTopic.title}".
Beschreibung: ${curatedTopic.description}

${GENDERING_INSTRUCTIONS}

${FAQ_STYLE_INSTRUCTIONS}

Zitierformat:
${citationRef}

Relevante Paragraphen:
${refText}

Antworte NUR mit validem JSON:
{
  "slug": "${curatedTopic.slug}",
  "title": "${curatedTopic.title}",
  "description": "...",
  "questions": [
    { "question": "...", "answer": "...", "references": [...] }
  ]
}`;
}
```

## State of the Art

| Old Approach | Current Approach | When Changed | Impact |
|--------------|------------------|--------------|--------|
| Single massive FAQ prompt | Per-topic focused prompts | Phase 04.3 (now) | Avoids token limit overflow, enables incremental regeneration |
| LLM-discovered topics | Curated topic list | Phase 04.3 (now) | Stable, reproducible FAQ content across regeneration cycles |
| ~200KB prompt with 100 topics | ~10-30KB prompt per topic | Phase 04.3 (now) | Each response well within 32K token limit |

**Root cause confirmed:** Debug output shows `outputTokens: 34923` against `maxOutputTokens: 32000`. The Claude CLI `--output-format json` wraps in `{"result":"..."}`, and the JSON extraction regex finds the inner `{...}` which starts at a truncation boundary (closing `}` fragment). This is not a `repairJSON` bug -- the content is genuinely incomplete.

## Open Questions

1. **Exact topic-to-summary matching strategy**
   - What we know: 1,759 unique labels, curated topics are broader categories
   - What's unclear: Whether keyword substring matching captures enough paragraphs without false positives
   - Recommendation: Implement substring matching first, log match counts per topic, adjust keywords if needed. A curated topic matching 0 paragraphs is a red flag.

2. **Whether to store per-topic JSON files**
   - What we know: CONTEXT.md suggests `data/llm/faq/topics/{slug}.json` for granular regeneration
   - What's unclear: Whether the complexity is worth it for 12-18 topics
   - Recommendation: Store per-topic files for debuggability and incremental regeneration, merge into `topics.json` at the end. Low cost, high benefit for development iteration.

3. **Whether `--topic` flag for single-topic regeneration adds value**
   - What we know: Per-topic architecture naturally supports it
   - What's unclear: How often it will be used in practice
   - Recommendation: Implement it -- low effort given per-topic architecture, useful for iteration during development.
