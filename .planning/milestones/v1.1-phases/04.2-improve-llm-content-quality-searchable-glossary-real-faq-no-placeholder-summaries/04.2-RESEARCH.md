# Phase 04.2: Improve LLM Content Quality - Research

**Researched:** 2026-03-11
**Domain:** LLM content pipeline, client-side filtering, content quality validation
**Confidence:** HIGH

## Summary

This phase is a content quality overhaul of the existing Phase 4 LLM infrastructure. All 23 summary files (9 Gemeindeordnungen + 14 Stadtrechte) currently have `placeholder: true` in their metadata, meaning summaries are auto-generated from titles, topics are all "Allgemeine Bestimmungen", FAQ questions are formulaic, glossary definitions lack proper BL coverage, and reference labels use raw keys like "Par. 35 burgenland" instead of proper Austrian legal citations.

The codebase is well-structured for this work: `scripts/llm-analyze.js` already has the full pipeline (summaries, FAQ, glossary) with Claude CLI integration via `spawnSync`, `scripts/generate-pages.js` reads from the same JSON format to render HTML, and the E2E tests validate the rendered output. The core work is: (1) improve LLM prompts for quality content, (2) split CLI commands, (3) add a validation script, (4) add glossary client-side search, and (5) regenerate all content.

**Primary recommendation:** Restructure `llm-analyze.js` into separate focused commands with improved prompts, add validation, add glossary filter UI, then run the full regeneration pipeline outside Claude Code session.

### Pattern 1: Separate CLI Entry Points
**What:** Each content type gets its own npm script mapping to different flags on llm-analyze.js
**When to use:** All LLM regeneration operations
**Example:**
```json
{
  "llm:summaries": "node scripts/llm-analyze.js --generate --force",
  "llm:faq": "node scripts/llm-analyze.js --faq --force",
  "llm:glossary": "node scripts/llm-analyze.js --glossary --force",
  "llm:all": "node scripts/llm-analyze.js --generate --force && node scripts/llm-analyze.js --faq --force && node scripts/llm-analyze.js --glossary --force",
  "llm:validate": "node scripts/llm-validate.js"
}
```

### Pattern 2: Glossary Client-Side Filter
**What:** Text input at top of glossary page that instantly filters terms by typing
**When to use:** Glossary page only
**Implementation approach:**
```javascript
// In generate-pages.js: add input element before A-Z nav
// In main.js: add initGlossaryFilter() function
function initGlossaryFilter() {
  const input = document.getElementById('glossar-filter');
  if (!input) return;

  input.addEventListener('input', () => {
    const query = input.value.toLowerCase().trim();
    document.querySelectorAll('[id^="letter-"]').forEach(section => {
      const parent = section.closest('.mb-8');
      if (!parent) return;
      let anyVisible = false;
      parent.querySelectorAll('[id]:not([id^="letter-"])').forEach(termDiv => {
        const term = termDiv.querySelector('h3')?.textContent?.toLowerCase() || '';
        const def = termDiv.querySelector('p')?.textContent?.toLowerCase() || '';
        const match = !query || term.includes(query) || def.includes(query);
        termDiv.style.display = match ? '' : 'none';
        if (match) anyVisible = true;
      });
      // Hide letter section header if no terms visible
      section.style.display = anyVisible ? '' : 'none';
      parent.style.display = anyVisible ? '' : 'none';
    });
  });
}
```

### Pattern 3: Validation Script (llm-validate.js)
**What:** Standalone validation that checks all LLM JSON for quality criteria
**Checks:**
- No `placeholder: true` in any meta
- Minimum content length for summaries (>50 chars)
- Proper German umlauts present (not ASCII-safe spellings)
- References span multiple BL (not just first 3)
- Citation format uses proper abbreviations (Bgld., Ktn., etc.)
- FAQ has real content (not formulaic "Was regelt..." pattern)
- Glossary terms have definitions >30 chars

### Pattern 4: Improved Prompts
**What:** Redesigned LLM prompts that produce natural, varied content
**Key changes from current prompts:**
- Summary prompt: Explicitly ban "Dieser Paragraph regelt..." opener, require varied sentence structures, require proper umlauts
- FAQ prompt: Feed ALL law paragraphs (not just topic samples), let Claude determine topics, require BL-specific answer structure
- Glossary prompt: Feed ALL law texts (not just 3 sample laws), require cross-BL references, require proper citations
- All prompts: Specify citation format map (burgenland -> "Bgld. GO", kaernten -> "Ktn. AGO", etc.)

### Anti-Patterns to Avoid
- **Incremental skip on regeneration:** The `--force` flag must delete existing files first. Current code has `existsSync` skip logic that must be bypassed.
- **Truncating law text in prompts:** Current glossary prompt only sends 3 laws with 5 paragraphs each (slice(0,3), slice(0,5)). Must send comprehensive content for cross-BL coverage.
- **Hardcoded topic list:** Current FAQ uses 10 hardcoded topics. Must let LLM determine topics from actual content analysis.

## Common Pitfalls

### Pitfall 1: Claude CLI Token Limits
**What goes wrong:** Large laws (e.g., Wien Stadtverfassung) have 100+ paragraphs. Sending all paragraph text in a single prompt may exceed context limits.
**Why it happens:** spawnSync sends entire prompt as stdin to `claude -p`
**How to avoid:** The current approach of one LLM pass per law is fine for most laws. Wien has 131 paragraphs -- monitor for timeout (current 300s limit). If a law fails, the script should log clearly and not silently fall back to placeholder.
**Warning signs:** spawnSync returning status !== 0, empty result

### Pitfall 2: JSON Parsing from LLM Output
**What goes wrong:** Claude may return markdown-wrapped JSON (```json ... ```) or include explanatory text before/after JSON.
**Why it happens:** LLM output format inconsistency
**How to avoid:** Current code already handles this with regex extraction (`content.match(/```json\s*([\s\S]*?)\s*```/)`). Keep this pattern. Add `--output-format json` to claude CLI call (already present).
**Warning signs:** JSON.parse errors in console

### Pitfall 3: Umlaut Consistency
**What goes wrong:** Prompts using ASCII-safe spellings (oe, ae, ue) may cause LLM to respond with ASCII-safe text.
**Why it happens:** Prompt text itself uses "oesterreichisches", "fuer", etc.
**How to avoid:** Use proper umlauts in the LLM prompts themselves. The prompts are sent via stdin, not shell, so umlauts are safe. Validation script should check for common ASCII-safe patterns like "fuer", "ueber", "oesterreich" in generated content.
**Warning signs:** Content containing "oe" instead of "oe" in user-facing text

### Pitfall 4: Running LLM Pipeline Inside Claude Code Session
**What goes wrong:** Claude CLI subprocess calls fail when invoked from within a Claude Code session (nested Claude invocation).
**Why it happens:** Claude CLI detects it's already in a session
**How to avoid:** Pipeline must be run manually by user outside Claude Code: `npm run llm:all`. The planner should create a task that prepares all code changes, then instructs the user to run regeneration separately.
**Warning signs:** `isClaudeAvailable()` returns true but `callClaude()` fails

### Pitfall 5: FAQ Reference Accuracy
**What goes wrong:** LLM invents paragraph numbers that don't exist in the actual law.
**Why it happens:** LLM hallucination on reference details
**How to avoid:** Feed actual paragraph numbers in the prompt context. Validation script should cross-check references against parsed law data.
**Warning signs:** Broken links on FAQ pages (href to #p999 that doesn't exist)

### Pitfall 6: Glossary Filter Breaking Pagefind
**What goes wrong:** Hiding terms with `display: none` might confuse Pagefind search results that link to glossary anchors.
**Why it happens:** Pagefind indexes at build time, filter is runtime only
**How to avoid:** Client-side filter is purely visual (display toggle). Pagefind results will still link correctly. Filter input should have `data-pagefind-ignore` to prevent indexing the filter itself.
**Warning signs:** None -- this is a non-issue since Pagefind indexes static HTML at build time

## Open Questions

1. **Token limits for large laws**
   - What we know: Wien Stadtverfassung has 131 paragraphs. Current timeout is 300s.
   - What's unclear: Whether all paragraph text fits in Claude CLI context window
   - Recommendation: Try full content first. If it fails, batch into chunks of ~50 paragraphs and merge results.

2. **Exact BL abbreviation format**
   - What we know: Austrian legal practice uses abbreviations like "Bgld.", "Ktn.", "NOe."
   - What's unclear: Whether "NOe." or "NOe" is more standard, same for "OOe." vs "OOe"
   - Recommendation: Use the format matching official short titles in config.js: Bgld., Ktn., NOe., OOe., Sbg., Stmk., Tir., Vbg., Wr. This matches the `name` field patterns already in LAWS config.

3. **FAQ topic count and depth**
   - What we know: User wants Claude to determine topics and question count
   - What's unclear: Optimal balance between comprehensiveness and page load
   - Recommendation: Let the prompt suggest 10-20 topics with 3-8 questions each. Validate output is reasonable.
