# Phase 02.1: Frontend Testing Infrastructure - Research

**Researched:** 2026-03-11
**Domain:** Playwright E2E testing for static Vite-built site
**Confidence:** HIGH

## Summary

This phase adds Playwright-based end-to-end tests to verify the 10 UAT scenarios from Phase 2. The project is a Vite 7 multi-page static site (ESM, TailwindCSS v4) deployed to GitHub Pages. Existing vitest unit tests cover Node.js scripts; Playwright will test the built HTML/JS/CSS in a real browser.

Playwright is the clear standard for this use case. Its built-in test runner, `webServer` config option (to auto-serve the built site), screenshot API, and `@axe-core/playwright` integration cover all requirements. The `vite preview` command serves the `dist/` output on a local port, making it the ideal `webServer.command` for Playwright.

**Primary recommendation:** Use `@playwright/test` with its own test runner (not vitest integration), Chromium-only for CI speed, `vite preview` as the webServer, and `@axe-core/playwright` for WCAG checks.

### Pattern 1: Playwright Config with webServer
**What:** Playwright auto-starts `vite preview` before tests and tears it down after.
**When to use:** Always -- this is the standard pattern for testing built static sites.
**Example:**
```javascript
// e2e/playwright.config.js
import { defineConfig, devices } from '@playwright/test';

export default defineConfig({
  testDir: './tests',
  outputDir: './test-results',
  fullyParallel: true,
  forbidOnly: !!process.env.CI,
  retries: process.env.CI ? 1 : 0,
  workers: process.env.CI ? 1 : undefined,

  use: {
    baseURL: 'http://localhost:4173',
    screenshot: 'on',
    trace: 'on-first-retry',
  },

  projects: [
    {
      name: 'desktop-chromium',
      use: { ...devices['Desktop Chrome'] },
    },
    {
      name: 'mobile',
      use: { ...devices['iPhone 13'], viewport: { width: 375, height: 812 } },
    },
  ],

  webServer: {
    command: 'npm run preview -- --port 4173',
    url: 'http://localhost:4173',
    reuseExistingServer: !process.env.CI,
    timeout: 30000,
  },
});
```

### Pattern 2: Screenshot Capture per Test
**What:** Each test captures a screenshot of the relevant UI state for visual review.
**When to use:** Every test -- this is a locked requirement.
**Example:**
```javascript
import { test, expect } from '@playwright/test';

test('ToC is collapsible', async ({ page }) => {
  await page.goto('/gemeindeordnung/gemeindeordnungen/wien.html');

  const details = page.locator('details').first();
  await expect(details).toBeVisible();

  // Capture collapsed state
  await page.screenshot({ path: 'e2e/screenshots/toc-collapsed.png', fullPage: false });

  // Expand
  await details.locator('summary').click();
  await expect(details).toHaveAttribute('open', '');

  // Capture expanded state
  await page.screenshot({ path: 'e2e/screenshots/toc-expanded.png', fullPage: false });
});
```

### Pattern 3: Accessibility Assertions with axe
**What:** Run axe-core against pages to verify WCAG 2.1 AA compliance.
**Example:**
```javascript
import { test, expect } from '@playwright/test';
import AxeBuilder from '@axe-core/playwright';

test('WCAG AA accessibility', async ({ page }) => {
  await page.goto('/gemeindeordnung/gemeindeordnungen/wien.html');

  const results = await new AxeBuilder({ page })
    .withTags(['wcag2a', 'wcag2aa', 'wcag21aa'])
    .analyze();

  expect(results.violations).toEqual([]);
});
```

### Anti-Patterns to Avoid
- **Testing against dev server instead of built output:** Dev server may differ from production (no minification, different asset paths). Always test against `vite build` + `vite preview`.
- **Hardcoding absolute URLs:** Use `baseURL` from config and relative paths in `page.goto()`.
- **Visual regression snapshots in CI:** Platform-dependent font rendering makes cross-platform snapshot comparison flaky. Use explicit screenshots for human review, not `toHaveScreenshot()` for automated comparison.
- **Testing all 9 Bundeslaender:** Redundant for E2E -- test 1-2 representative pages. All pages use the same template.

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Serving built site for tests | Custom express server | `vite preview` via webServer config | Already configured, handles base path |
| Accessibility checks | Manual color contrast assertions | @axe-core/playwright | Covers 50+ rule categories, maintained by Deque |
| Mobile viewport simulation | Manual resize | Playwright `devices` presets | Accurate device emulation including UA, touch events |
| Waiting for page ready | sleep/setTimeout | Playwright auto-waiting + explicit `waitForSelector` | Built-in auto-retry, more reliable |
| CI browser install | Manual apt-get | `npx playwright install chromium --with-deps` | Handles all OS-specific dependencies |

## Common Pitfalls

### Pitfall 1: Base Path Mismatch
**What goes wrong:** Vite config has `base: '/gemeindeordnung/'` but tests navigate to `/` or `/wien.html`.
**Why it happens:** Vite preview serves the built site under the base path.
**How to avoid:** Set `baseURL: 'http://localhost:4173/gemeindeordnung'` in Playwright config, or use full paths in goto.
**Warning signs:** 404 errors on all page navigations.

### Pitfall 2: Generated HTML Not Present
**What goes wrong:** Tests fail because `src/gemeindeordnungen/*.html` don't exist yet (they are generated by `scripts/generate-pages.js`).
**Why it happens:** The build pipeline is: fetch -> parse -> generate -> vite build. Tests need all steps.
**How to avoid:** In CI, run the full pipeline before tests. Locally, ensure pages are generated. The `vite build` step already requires them.
**Warning signs:** Vite build fails with empty input discovery.

### Pitfall 3: Clipboard API in CI
**What goes wrong:** `navigator.clipboard.writeText()` fails in headless browser / insecure context.
**Why it happens:** Clipboard API requires secure context (HTTPS) or explicit permission.
**How to avoid:** Grant clipboard permission in browser context: `context.grantPermissions(['clipboard-read', 'clipboard-write'])`. The code already has a fallback via `execCommand('copy')`.
**Warning signs:** Copy-link test fails only in CI, works locally.

### Pitfall 4: Scroll-to-Top Visibility Timing
**What goes wrong:** Test checks for scroll-to-top button but it hasn't appeared yet.
**Why it happens:** Button shows after scrolling past 300px, triggered by scroll event listener with passive flag.
**How to avoid:** After `page.evaluate(() => window.scrollTo(0, 500))`, wait for button visibility: `await expect(btn).toBeVisible()`.

### Pitfall 5: CI Runner Has No Browsers
**What goes wrong:** `npx playwright test` fails in CI with "browser not found."
**Why it happens:** Playwright browsers must be explicitly installed in CI.
**How to avoid:** Add `npx playwright install chromium --with-deps` step in GitHub Actions before running tests.

## Code Examples

### GitHub Actions Integration
```yaml
# Add these steps to .github/workflows/deploy.yml BEFORE upload-pages-artifact

- name: Install Playwright browsers
  run: npx playwright install chromium --with-deps

- name: Run E2E tests
  run: npx playwright test --config=e2e/playwright.config.js

- name: Upload test screenshots
  if: always()
  uses: actions/upload-artifact@v4
  with:
    name: playwright-screenshots
    path: e2e/screenshots/
    retention-days: 14

- name: Upload test results
  if: failure()
  uses: actions/upload-artifact@v4
  with:
    name: playwright-report
    path: e2e/test-results/
    retention-days: 7
```

### npm Scripts Addition
```json
{
  "scripts": {
    "test:e2e": "playwright test --config=e2e/playwright.config.js",
    "test:e2e:ui": "playwright test --config=e2e/playwright.config.js --ui"
  }
}
```

### Representative Test: Anchor Highlight
```javascript
import { test, expect } from '@playwright/test';

test('anchor highlight on deep link', async ({ page }) => {
  await page.goto('/gemeindeordnung/gemeindeordnungen/wien.html#p1');

  const target = page.locator('#p1');
  await expect(target).toBeVisible();

  // Check highlight class was added
  await expect(target).toHaveClass(/anchor-highlight/);

  await page.screenshot({
    path: 'e2e/screenshots/anchor-highlight.png',
    clip: await target.boundingBox() || undefined,
  });

  // Wait for highlight to fade (2s timeout in main.js)
  await page.waitForTimeout(2100);
  await expect(target).not.toHaveClass(/anchor-highlight/);
});
```

## Open Questions

1. **Full build time in CI**
   - What we know: 9 Bundeslaender + Stadtrechte are fetched and parsed at build time. CI does full fetch from RIS API.
   - What's unclear: How long does the full pipeline take? If >3min, may need subset approach.
   - Recommendation: Start with full build. If CI is too slow, add a test-specific build script that uses only pre-committed fixture data (1-2 Bundeslaender).

2. **Screenshot storage strategy**
   - What we know: Screenshots should be CI artifacts. User wants visual verification.
   - What's unclear: Commit screenshots to repo (baseline reference) or only store as CI artifacts?
   - Recommendation: Store only as CI artifacts (gitignore `e2e/screenshots/`). No visual regression comparison -- screenshots are for human review.
