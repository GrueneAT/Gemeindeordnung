<decisions>
## Implementation Decisions

### Summary Regeneration
- Regenerate all 9 Bundesländer + all Stadtrechte from scratch (discard all placeholder data)
- No repetitive openers — varied, natural plain-language explanations. Not "Dieser Paragraph regelt..." every time
- Tone: sachlich-verständlich, nice to read, not formulaic
- Regenerate topic tags alongside summaries for better granularity (current "Allgemeine Bestimmungen" for everything is too broad)
- One LLM pass per law: summaries + topics together

### FAQ Content
- Regenerate the entire topic list — Claude picks topics based on actual law content
- Answer structure: Clear answer first, then Bundesländer-specific details as supplementary info. Users want to find how it works in their BL quickly; the comparison is a nice-to-have addition
- As many questions per topic as Claude deems necessary (no fixed count)
- References cover all relevant BL + Stadtrechte where provisions exist
- Proper legal citation format: "§ 15 Bgld. GO" instead of "Par. 15 burgenland"
- No Bundesland filter on FAQ pages — clear structure is sufficient
- Proper German umlauts in all content

### Glossary
- Full regeneration with references across all 9 BL + Stadtrechte
- Claude determines appropriate number of terms (not fixed at 20)
- Add client-side filter/search input at top of glossary page (instant text filtering)
- Proper German umlauts in definitions and all text
- Consistent citation format matching FAQ: "§ 15 Bgld. GO"

### LLM Pipeline
- Separate commands: `npm run llm:summaries`, `npm run llm:faq`, `npm run llm:glossary`, `npm run llm:all`
- Fix existing pipeline (llm-analyze.js) with proper prompts instead of placeholder generation
- Run outside Claude Code session (Claude CLI subprocess approach)
- Always overwrite existing data (no incremental skip — full quality reset)
- Add `npm run llm:validate` step: checks no placeholder flag, minimum content length, proper umlauts, references span multiple BL

### Claude's Discretion
- Exact prompt design for each content type
- Number of FAQ topics and questions per topic
- Number of glossary terms
- Abbreviation format for BL citations (Bgld., Ktn., NÖ, OÖ, Sbg., Stmk., Tir., Vbg., Wien)
- How to structure FAQ answers with BL supplementary info (paragraphs, bullet lists, etc.)
- Glossary client-side filter implementation (vanilla JS, same pattern as existing search)

</decisions>

<specifics>
## Specific Ideas

- FAQ answers should prioritize answering the question, not comparing BL — "somebody can find out quickly how it works in their BL, the comparison is a nice to have in addition"
- Summary text should be nice to read, not too repetitive — vary sentence structures and openings
- Reference labels should use proper Austrian legal citation style
- All user-facing text must use proper German umlauts (Phase 4.1 decision carries forward)
- Topics and FAQ themes should be consistent (regenerated from same content analysis)

</specifics>

## Existing Code Insights

### Reusable Assets
- `scripts/llm-analyze.js`: Existing pipeline script with Claude CLI invocation via spawnSync + stdin
- `scripts/generate-pages.js`: Page generator handles FAQ, glossary, and law pages — reads LLM JSON data
- `scripts/config.js`: 23-law registry with all metadata (categories, keys, display names)
- `src/js/main.js`: Existing JS with tooltip, collapsible, and filter functionality
- `src/glossar.html`: Existing glossary page with A-Z jump links and data-pagefind-body

### Established Patterns
- LLM data stored in `data/llm/{summaries,faq,glossary}/` as JSON
- Summary JSON: `{meta: {}, paragraphs: {num: {summary, topics}}}` per law
- FAQ JSON: `{meta: {}, topics: [{slug, title, description, questions: [{question, answer, references}]}]}`
- Glossary JSON: `{meta: {}, terms: [{term, slug, definition, references}]}`
- `placeholder: true` flag in meta signals content needs regeneration
- Pagefind indexes all `data-pagefind-body` content automatically

### Integration Points
- `generate-pages.js` reads JSON and renders HTML — no changes needed to rendering if JSON structure stays same
- Glossary page needs new filter input + JS for client-side filtering
- FAQ HTML pages regenerated from topics.json by generate-pages.js
- Inline glossary tooltips in law pages read from terms.json
