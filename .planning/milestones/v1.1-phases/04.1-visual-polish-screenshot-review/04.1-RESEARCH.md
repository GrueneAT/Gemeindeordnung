### Pattern 1: Screenshot-Driven Visual QA Cycle
**What:** Build site, capture screenshots via Playwright, read PNGs with Claude's vision, identify issues, fix code, repeat.
**When to use:** Every time UI code changes.
**Cycle:**
```bash
# 1. Build + index
npm run build && npx pagefind --site dist --force-language de

# 2. Capture screenshots
npx playwright test --config=e2e/playwright.config.js --project=desktop-chromium

# 3. Read and review each screenshot (Claude vision)
# 4. Fix issues in src/css/main.css, scripts/generate-pages.js, or src/assets/
# 5. Regenerate pages if template changed: node scripts/generate-pages.js
# 6. Repeat from step 1
```

## Common Pitfalls

### Pitfall 1: Editing Generated Files Instead of Templates
**What goes wrong:** Changes to `src/gemeindeordnungen/*.html` are lost on next `generate-pages.js` run.
**Why it happens:** Easy to forget these are generated files.
**How to avoid:** Always trace back to `scripts/generate-pages.js` for any HTML structure changes. CSS changes go in `src/css/main.css`.
**Warning signs:** Changes disappear after running `node scripts/generate-pages.js`.

### Pitfall 2: Forgetting Pagefind Rebuild Before Screenshots
**What goes wrong:** Search-related screenshots fail or show stale data.
**Why it happens:** Pagefind index lives in `dist/` and must be rebuilt after `npm run build`.
**How to avoid:** Always use the full pipeline: `npm run build && npx pagefind --site dist --force-language de`.
**Warning signs:** Search tests timeout or show no results.

### Pitfall 3: German Umlaut Encoding in Templates
**What goes wrong:** Text shows "oesterreichischen" instead of "osterreichischen" or garbled umlauts.
**Why it happens:** The current template uses ASCII-safe spellings (e.g., "oesterreichischen Bundeslaender", "Hauptstueck"). This appears to be intentional to avoid encoding issues in the generation pipeline.
**How to avoid:** Check whether switching to proper umlauts (o-umlaut, u-umlaut, etc.) is safe by verifying the full pipeline renders them correctly. All HTML pages have `<meta charset="UTF-8">`.
**Warning signs:** Garbled characters in screenshots, or inconsistency between template text and parsed law text (which already uses umlauts).

### Pitfall 4: Mobile Viewport Testing Nuance
**What goes wrong:** Mobile tests run in desktop-chromium project with explicit viewport resize, not a true mobile browser.
**Why it happens:** WebKit is not installed in the environment.
**How to avoid:** Use `page.setViewportSize({ width: 375, height: 812 })` within desktop-chromium project. The existing mobile.spec.js already does this correctly.
**Warning signs:** Mobile-specific CSS (hover media queries, touch targets) may not behave identically to real mobile.

### Pitfall 5: Logo SVG ViewBox Mismatch
**What goes wrong:** Logo renders at wrong aspect ratio or gets clipped in the 32x32px container.
**Why it happens:** SVG viewBox dimensions don't match the container aspect ratio.
**How to avoid:** Ensure the replacement SVG has a square or near-square viewBox that works at `w-8 h-8` (32x32px). Test by viewing the browse-page-wien.png screenshot after replacement.
**Warning signs:** Logo appears stretched, squished, or clipped in header.

## Gruene Logo Research

### Current State
The file `src/assets/gruene-logo.svg` contains a hand-drawn sunflower-inspired logo with 8 petals in `#88b828` and a center circle in `#005538`. This is incorrect.

### Correct Logo
The Austrian Gruene party logo is a stylized green "G" that incorporates an arrow element, giving it a dynamic, forward-pointing quality. It is NOT the German Gruene sunflower. Key characteristics:
- Single green color mark (no text in the logo mark itself)
- Organic brushstroke style
- Uppercase "G" shape with integrated arrow
- The party name "Die Gruenen" and URL "gruene.at" appear as text alongside the mark, not within the SVG mark itself

### Logo Color Reference
From the official gruene.at website: primary green is `#73A303` (or nearby). The project theme already uses `#6BA539` for `gruene-green` and `#005538` for `gruene-dark`. The logo should use the project's `gruene-green` or similar to maintain consistency.

### Sourcing Options (Claude's Discretion)
1. **Wikimedia Commons:** `Gruene_Logo.svg` file exists (2012 version) -- may be outdated
2. **bildgenerator.gruene.at:** Has `/resources/images/gruene-logo.svg` but it turned out to be the German party logo
3. **Create accurate SVG:** Based on the known design (green arrow-G), create a clean SVG that matches the brand mark. This is the most reliable approach since web sources have mixed up Austrian and German logos.
4. **Fetch from gruene.at directly:** The site uses the logo but extracting programmatically may be complex

**Recommendation:** Create a clean, accurate SVG of the Austrian Gruene arrow-G logo mark. The design is well-documented (stylized G with arrow), and creating it ensures correct brand representation without depending on potentially mixed-up external sources.
