# Phase 6: Search-Hero Homepage & Navigation - Research

**Researched:** 2026-03-12
**Domain:** Homepage hero layout, discovery link generation, header navigation, responsive design
**Confidence:** HIGH

## Summary

This phase transforms the homepage from a card-grid document browser into a "search first" experience with a prominent hero search bar, quick-access discovery links (FAQ topics and glossary terms), and polished cross-page navigation. The technical work is primarily in `scripts/generate-pages.js` (rewriting `generateIndexPage()` and refining `generateHeader()`), `src/css/main.css` (hero section styles, discovery chip styles, collapsible card grid), and `src/js/search.js` (hero search input initialization and sync with header search).

The existing codebase provides strong foundations: `generateIndexPage()` already receives `lawsByCategory` and `faqTopics` data, the header is generated by a shared `generateHeader()` function used across all pages, and the search module exports `initSearch()` which binds to `#search-input`. The main challenges are: (1) having two search inputs (hero + header) that share one Pagefind instance, (2) making the mobile nav accommodate FAQ/Glossar links without crowding, and (3) reading glossary terms at build time for discovery links (glossary data is in `data/llm/glossary/terms.json` but this file may not exist yet -- the glossary page generator reads it from the same path).

**Primary recommendation:** Split into three plans: (1) Hero section layout + discovery links in generate-pages.js, (2) Header navigation polish + mobile nav across all pages, (3) Hero search integration in search.js + E2E tests for all new UI.

### Pattern 1: Hero Section with Dual Search Input
**What:** Homepage has a large hero search input (`#hero-search-input`) that shares the Pagefind instance with the header search input (`#search-input`).
**When to use:** Index page only.

**Recommended approach -- shared state via search.js:**
```javascript
// In search.js initSearch():
const heroInput = document.getElementById('hero-search-input');
if (heroInput) {
  // Hero input mirrors header input
  heroInput.addEventListener('input', (e) => {
    const query = e.target.value.trim();
    currentQuery = query;
    // Sync to header input
    if (searchInput) searchInput.value = query;
    handleSearchInput(e);
  });
  // Optionally: results appear in a panel below hero, not header dropdown
  // Use a separate dropdown container for hero: #hero-search-dropdown
}
```

**Key decision:** Where do hero search results appear?
- CONTEXT.md says: "Hero search results appear in the Phase 5 expanded panel directly below the hero input"
- This means: the hero section needs its own results container positioned below the hero input
- The header search dropdown should be hidden/suppressed when hero search is active
- On scroll or on non-index pages, only header search is active

**HTML structure for hero:**
```html
<section class="hero-section bg-gruene-light py-12 sm:py-16 text-center" data-pagefind-ignore>
  <h1 class="text-3xl sm:text-4xl font-bold text-gruene-dark mb-3">
    Gemeindeordnungen der oesterreichischen Bundeslaender
  </h1>
  <p class="text-lg text-gruene-dark/80 mb-8">
    Alle Gemeindeordnungen und Stadtrechte durchsuchen
  </p>
  <div class="hero-search-container max-w-2xl mx-auto px-4">
    <div class="relative">
      <input id="hero-search-input" type="search" minlength="3" autocomplete="off"
        placeholder="Gesetz, Thema oder Begriff suchen..."
        class="w-full border-2 border-gruene-green/50 rounded-xl pl-12 pr-4 py-4 text-lg bg-white text-gruene-dark
               focus:outline-none focus:ring-2 focus:ring-gruene-green focus:border-gruene-green shadow-lg" />
```

### Pattern 2: Discovery Links (Build-Time Static)
**What:** FAQ topic chips and glossary term chips generated at build time from JSON data files.
**When to use:** Index page below hero section.

**Data sources:**
- FAQ topics: `data/llm/faq/curated-topics.json` -- has 15 topics with `slug`, `title`, `description`
- Glossary terms: `data/llm/glossary/terms.json` -- may not exist in all environments; read at build time if available

**Implementation in generate-pages.js:**
```javascript
// Already receives faqTopics in generateIndexPage(lawsByCategory, faqTopics)
// Select first 8 topics for discovery chips
const discoveryFaqTopics = faqTopics.slice(0, 8);
const faqChips = discoveryFaqTopics.map(t =>
  `<a href="faq/${t.slug}.html" class="discovery-chip">${escapeHtml(t.title)}</a>`
).join('\n');

// Glossary terms: read from terms.json, pick 8 most useful
const glossaryPath = join(rootDir, 'data', 'llm', 'glossary', 'terms.json');
let glossaryChips = '';
if (existsSync(glossaryPath)) {
  const terms = JSON.parse(readFileSync(glossaryPath, 'utf-8')).terms || [];
  const selected = terms.slice(0, 8); // or hand-pick important ones
  glossaryChips = selected.map(t =>
    `<a href="glossar.html#${t.slug}" class="discovery-chip">${escapeHtml(t.term)}</a>`
  ).join('\n');
}
```

**Note on glossary data availability:** The glossary directory exists at `data/llm/glossary/` but `terms.json` is currently empty/missing. The generator already handles this gracefully (falls back to empty array). If glossary terms are not available at build time, the glossary discovery section should be omitted or show a placeholder. The code must handle the missing file case.

### Pattern 4: Mobile Navigation
**What:** FAQ and Glossar links visible on mobile without hamburger menu.
**When to use:** All pages, header navigation.

**Current state:** `generateHeader()` has nav links wrapped in `hidden sm:flex`, making them invisible on mobile. The mobile overlay only handles search, not navigation.

**Recommended approach -- visible compact nav row:**
```html
<!-- In header, replace hidden sm:flex with always-visible compact links -->
<nav class="flex items-center gap-2 text-xs sm:text-sm shrink-0 sm:ml-4" data-pagefind-ignore>
  <a href="${prefix}faq/index.html" class="text-gruene-dark hover:underline">FAQ</a>
  <span class="text-gray-300">|</span>
  <a href="${prefix}glossar.html" class="text-gruene-dark hover:underline">Glossar</a>
</nav>
```

This avoids hamburger menu complexity. FAQ and Glossar are only 2 links -- they can fit as small text links even on 375px.

**Header layout on mobile:**
- Row 1: Logo + site title + FAQ|Glossar links (compact)
- Row 2: Search input (full width)
- This matches the existing pattern where search wraps below logo on mobile

### Anti-Patterns to Avoid
- **Don't create a separate search page:** Results must appear inline below hero input, not navigate away
- **Don't duplicate Pagefind initialization:** One `loadPagefind()` call, shared by both inputs
- **Don't hide the card grid entirely:** Users browsing by Bundesland still need it
- **Don't use hamburger menu for 2 links:** Hamburger adds complexity for minimal content; use visible compact links
- **Don't break existing search functionality:** Header search on law pages must continue working exactly as before

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Collapsible sections | Custom JS toggle | HTML `<details>/<summary>` | Native, accessible, no JS required |
| Discovery link data | Hardcoded HTML arrays | Read from curated-topics.json at build | Single source of truth, auto-updates |
| Mobile nav toggle | Custom hamburger menu | Always-visible compact text links | Only 2 links, hamburger is overkill |
| Search input sync | Complex event bus | Direct DOM value sync in search.js | Simple, no state management needed |

**Key insight:** This phase is primarily a layout/template change with minimal new JavaScript logic. The heavy lifting (Pagefind, search rendering, mobile overlay) already exists from Phases 3 and 5.

## Common Pitfalls

### Pitfall 1: Dual Search Input Event Conflicts
**What goes wrong:** Both hero and header search inputs fire events, causing double searches or race conditions.
**Why it happens:** `initSearch()` binds to `#search-input` only; adding `#hero-search-input` needs careful event handling.
**How to avoid:** In `initSearch()`, detect hero input and wire it up. When hero input is active, suppress header dropdown. When user scrolls past hero, focus shifts to header search.
**Warning signs:** Search results appearing in both dropdowns, or flickering between them.

### Pitfall 2: Hero Search Results Panel Positioning
**What goes wrong:** The Phase 5 expanded results panel is positioned relative to the header search container. On the hero, it needs to be positioned relative to the hero container.
**Why it happens:** CSS `.search-dropdown` uses `position: absolute; top: 100%` relative to `.search-container`.
**How to avoid:** Give hero its own `.hero-search-container` with its own dropdown div. `search.js` must target the correct dropdown based on which input is active.
**Warning signs:** Results panel appearing at wrong position, behind hero, or at top of page.

### Pitfall 3: Mobile Overlay Assumes Header-Resident Input
**What goes wrong:** The mobile overlay in `search.js` saves/restores `searchInput`/`searchDropdown` refs assuming they point to header elements. If hero input is the active one, restore breaks.
**Why it happens:** `openMobileOverlay()` stores desktop refs and replaces them with mobile overlay refs. On close, it restores the original refs. If the original was the hero input, this needs to work correctly.
**How to avoid:** Track which context (hero vs header) was active before overlay opened. On close, restore to the correct context. STATE.md already flags this concern: "Mobile overlay assumes header-resident search input -- Phase 6 hero refactor must verify overlay still works."
**Warning signs:** After closing mobile overlay on index page, search input stops working.

### Pitfall 4: Glossary Terms Data Missing
**What goes wrong:** `data/llm/glossary/terms.json` does not exist (directory is empty). Build fails or glossary discovery section is empty.
**Why it happens:** Glossary generation may not have been run, or terms were not committed.
**How to avoid:** Always check `existsSync()` before reading. If missing, omit the glossary discovery section gracefully. Do not fail the build.
**Warning signs:** Empty "Glossar" section on homepage with no chips.

### Pitfall 5: Existing E2E Tests Break
**What goes wrong:** `card-grid.spec.js` expects the current index page layout (card grid as primary element). Changing to hero layout breaks assertions.
**Why it happens:** Tests check for specific selectors, grid visibility, and card positions that change with the hero redesign.
**How to avoid:** Update `card-grid.spec.js` to expect the new layout (cards inside collapsible section). Add new hero-specific test specs. The `card-grid-index.png` screenshot will need updating.
**Warning signs:** Multiple E2E failures in card-grid and mobile tests after layout change.
